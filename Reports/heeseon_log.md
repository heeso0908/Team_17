### 1. 데이터 구조 및 이벤트 기준 정리
스타벅스 CRM 데이터의 이벤트 구조를 다시 점검하고, 분석 기준을 정리했다.

#### 주요 이벤트
- `received` : 오퍼 발송
- `viewed` : 오퍼 열람
- `completed` : 오퍼 완료
- `transaction` : 실제 구매

#### 정리한 기준
- 오퍼 성과 분석은 `received → viewed → completed` 퍼널 기준으로 진행
- 매출 분석은 `transaction` 기준으로 별도 진행
- `informational` 오퍼는 일반 프로모션 오퍼와 동일한 완료 개념으로 해석하지 않음

---

### 2. 오퍼 퍼널 지표 계산 및 시각화
전체 오퍼 진행 현황을 보기 위해 퍼널 구조를 집계하고 시각화했다.

#### 계산 지표
- 총 발송 수
- 총 열람 수
- 총 완료 수
- 발송 → 열람 전환율
- 열람 → 완료 전환율

#### 시각화 정리
보고용 가독성을 위해 x축 이벤트명을 한글로 변경했다.

- `received` → 발송
- `viewed` → 열람
- `completed` → 완료

---

### 3. 매출 KPI 정리
구매 이벤트를 기준으로 핵심 매출 지표를 계산했다.

#### 주요 KPI
- 총 매출
- 총 거래 건수
- 객단가(AOV)
- 전체 고객 수
- 구매 고객 수
- ARPU
- ARPPU
- 평균 구매 빈도

#### 개념 정리
- **AOV** : 거래 1건당 평균 구매 금액
- **ARPU** : 전체 고객 1명당 평균 매출
- **ARPPU** : 구매 고객 1명당 평균 매출

### 4. 고객 구매 행동 탐색
고객 단위로 구매 패턴을 집계해 분포를 확인했다.

#### 고객별 집계 항목
- 총 구매 금액
- 구매 횟수
- 고객별 평균 객단가

#### 확인한 내용
- 고객별 총 구매 금액 분포
- 고객별 구매 횟수 분포
- 고객별 평균 객단가 분포
- 상위 구매 고객 TOP 10

### 5. 오퍼 완료 고객 vs 미완료 고객 비교
오퍼 완료 여부에 따라 실제 구매 행동 차이가 있는지 비교했다.

#### 비교 지표
- 평균 총 구매 금액
- 평균 구매 횟수
- 평균 객단가

#### 해석
오퍼 성과는 완료율만으로 보기보다, 실제 매출과 구매 행동까지 연결해서 해석해야 한다는 방향을 잡았다.

## 2026-03-24

## 1. 오퍼 인스턴스 단위 데이터 구조 설계

분석의 기준 단위를 오퍼 인스턴스로 정의하고, 데이터셋 구성 방식을 재정립했다.

#### 핵심 설계 결정

- 같은 고객이 동일한 오퍼를 여러 번 받을 수 있으므로, 각 오퍼 수신 시점별로 `(received → viewed → completed)` 세트를 별도 인스턴스로 구분
- `customer_id` + `offer_id` + `time` 기준으로 그룹화하여 집계

#### 전환 유형 분리

- **전략적 전환** : 오퍼를 열람한 후 의도적으로 완료한 케이스
- **우연적 전환** : 오퍼를 열람하지 않고 완료된 케이스 (viewed 없이 completed)
- 초기 계산 시 두 유형이 혼재되어 전환율 100% 오류 발생 → 분리 후 수정

---

## 2. 오퍼 유형별 퍼널 분석

오퍼 유형에 따라 퍼널 구조를 세분화하여 분석했다.

#### 퍼널 1 — 3단계 완전 통과 오퍼

- 대상: bogo, discount 중 `received → viewed → completed` 3단계를 모두 거친 오퍼
- `informational` 오퍼는 completed 개념이 없으므로 제외

#### 퍼널 2 — 뷰 없이 완료된 오퍼

- 대상: bogo, discount 중 viewed 단계 없이 completed가 발생한 케이스
- 해석: 낭비된 리워드 (고객이 쿠폰을 확인하지 않고 결제)
- 추가 분석 필요

#### 퍼널 3 — informational 오퍼 영향 분석

- `received → viewed` 흐름만 존재하는 오퍼 유형
- 열람률 : 약 **65%**
- 24시간 내 구매 전환율 : 약 **40%**
- 인사이트 : 열람한 고객 vs 열람하지 않은 고객 비교 시, 구매 건수는 역행하지만 객단가·구매 금액은 상승하는 경향 확인

---

## 3. 보상형 오퍼 vs informational 오퍼 매출 비교

오퍼 유형에 따른 실질적인 매출 기여 차이를 비교했다.

#### 비교 방식

- 보상형 오퍼(bogo, discount) 완료 고객의 평균 결제액
- vs. informational 오퍼만 받은 고객의 평균 결제액

#### 결과 및 이슈

- 매출 증대율이 약 **1,900%** 수준으로 나타남 → 수치가 비현실적으로 크다는 문제 제기
- 지표 정의 재검토 필요: "보상형 오퍼를 받은 고객이 informational만 받은 고객 대비 얼마나 더 지출했는가"로 해석 기준 정립
- 계산 로직 재검토 및 개선 방향 논의 예정

## 1. 퍼널 구조 3가지 확정 및 코드 검증

분석 프레임워크의 핵심인 퍼널 3가지 구조를 최종 정의하고, 기존 코드의 정합성을 검증했다.

#### 확정된 퍼널 구조

- **퍼널 1** : `received → viewed → completed` — 정상적인 오퍼 사용 흐름 (전략적 전환)
- **퍼널 2** : `received → completed` — 쿠폰 미확인 결제 (낭비된 리워드)
- **퍼널 3** : informational 오퍼 조회 후 24시간 내 일반 거래 (오퍼 영향력 측정)

#### 코드 검증 작업

- 월요일에 수정한 8개 그래프 코드 중 일부 오류 가능성 발견
- merge 파일 기준으로 새로 작성한 코드와 기존 8개 코드 간 불일치 확인 필요
- ARPU(평균객단가) 계산 코드 오류 발견 → 수정 및 나머지 그래프 적용 여부 확인
- 피그마 문서의 "확인 필요" 표시된 부분 수정 작업 진행

#### 오전 목표 및 타임라인

- **오전**: 8개 그래프 코드 검증 완료
- **오늘**: 시각화 작업 전체 취합 + 통계 분석 완료

## 2. viewed 데이터 중복 처리 기준 결정

오퍼 열람 데이터의 중복 처리 방식을 팀 논의를 통해 확정했다.

#### 문제 상황

- 한 고객이 동일한 오퍼를 여러 번 조회한 경우 처리 방식 미정
- 중복 포함 시 열람 수: 약 **46,000건** → 과도한 수치
- 중복 포함 시 퍼널 전환율 및 집계가 왜곡될 수 있음

#### 결정 사항

- `person + offer_id` 기준으로 **최초 조회 시점(minimum time)만 남기기**로 확정
- 근거: 고객이 오퍼를 처음 열람한 반응이 핵심 지표 / 재조회는 내용을 잊고 다시 본 것으로 해석
- 중복 제거 후 수정된 데이터로 퍼널 전환율 그래프 재작성 예정

#### 데이터 컬럼 정의 명확화

- **Offer ID**: bogo_1, bogo_2, discount_1 등 세분화된 개별 오퍼 건 (총 10개)
- **Offer Label(Type)**: discount, bogo 등 오퍼 유형
- 두 컬럼의 의미 혼동 해소 → 이후 분석 시 명확히 구분하여 사용

## 3. 전처리 계획 수립

중복 처리 결정 이후 전처리 우선순위를 정립했다.

#### 전처리 순서

1. **1순위**: `merged_df` 전처리 파일에서 중복 데이터 제거
2. **2순위**: 별도의 transactions 테이블 생성 (일반 거래 + completed 연결된 거래만 포함)

#### 리스크 인지

- 데이터가 타임스탬프 절대시간 구조가 아닌 상대시간 구조이므로 분석 정확도 제한 가능성 존재
- 이를 리스크로 명시하고 해석 시 주의하기로 합의

## 4. 매출 데이터 중복 합산 오류 발견 및 해결

팀원이 bogo와 디스카운트 중복 합산 오류를 발견하여 전처리 구조를 전면 재설계했다.

#### 발견된 문제

- 일반 결제 + discount + bogo를 단순 합산하면 **총 매출을 초과**하는 값 발생
- 원인: 동일 거래가 discount와 bogo 두 조건을 동시에 만족하는 경우 **중복 합산** (~6만 달러 오차)

#### 해결 방안 — 카테고리 재분류

| 카테고리 | 정의 |
| --- | --- |
| 일반결제 | 오퍼 미적용 일반 거래 |
| bogo 단독 | bogo 조건만 만족하는 거래 |
| discount 단독 | discount 조건만 만족하는 거래 |
| 다중오퍼 | 두 조건을 동시에 만족하는 거래 |
- 4개 카테고리 합산 = 총 매출 100% (정확히 일치)
- **전처리 순서 조정**: 다중오퍼 항목 먼저 제거 → 이후 discount_1~4 분리
- 최종 항목 수: informational 제외 총 **10개 항목**

#### 시각화 팀 대응 계획

- 전처리 파일 완료 전까지 **퍼널 1번, 2번 코드 작성** 집중
- 퍼널 3번(Informational)은 매출 데이터가 없어 기존 8개 그래프 형식 미적용
- 전처리 완료 후 공유받아 나머지 작업 연결
